问题1:
什么是前后端分离？
概念
前后端分离是一种 Web 应用架构模式：前端（浏览器/App）和后端（服务器）完全解耦，二者只通过 HTTP API 通信，各自独立开发、独立部署。
传统模式（不分离）里，后端既写业务逻辑，也负责渲染 HTML 页面（如 JSP、PHP 模板），前端只是"被服务端生成的 HTML"。而前后端分离后，后端只提供 JSON 数据接口，前端自己负责渲染页面。

问题2:
历史对话的存储与传递

这个项目用了两套并行的 SQLite 存储，分工不同。
第一套：业务数据库（sever.db）—— 负责展示
用 SQLAlchemy ORM 定义了两张表：

ChatSessionTable：一行代表一次对话，记录 session_id、用户、标题、创建时间
ChatMessageTable：一行代表一条消息，记录 role（user/assistant/system）、content、反馈状态
每次对话时手动调用 append_message2db() 写入：

python
复制
append_message2db(session_id, "user", content)       # 用户发问时写
append_message2db(session_id, "assistant", assistant_message)  # 大模型回完写
这套数据的消费者是前端——通过 /v1/chat/get 接口拉取历史消息渲染到页面，以及支持点赞/点踩反馈。它不直接参与大模型调用。

第二套：Session 数据库（conversations.db）—— 负责上下文
使用 openai-agents SDK 提供的 AdvancedSQLiteSession：

python
复制
session = AdvancedSQLiteSession(
    session_id=session_id,
    db_path="./assert/conversations.db",
    create_tables=True
)
result = Runner.run_streamed(agent, input=content, session=session)
它的工作原理是：把每次对话的 messages（user + assistant 轮次）自动追加到 SQLite，下次调用时自动读出拼成完整的 messages 数组，按 OpenAI 格式传给大模型：

[
  {"role": "system",    "content": "你是小呆助手..."},
  {"role": "user",      "content": "第1轮问题"},
  {"role": "assistant", "content": "第1轮回答"},
  {"role": "user",      "content": "第2轮问题"},   ← 本次新增
]
这一切由 SDK 自动完成，业务代码只需传入同一个 session_id，无需手动拼 messages
